# AGENTS.md

Guidance for AI coding agents (and humans) working in this repository. This
file documents what the project _is_, how it's organized, how it's
implemented, what's vendored, how to build/test it, and non-obvious findings
uncovered while building it — so you don't have to rediscover them.

For user-facing docs (features, install, API, examples) see the READMEs
instead: [`README.md`](README.md) (project), [`crates/mlex/README.md`](crates/mlex/README.md)
(Rust crate), [`crates/mlex-node/README.md`](crates/mlex-node/README.md) (internal
Node glue crate), [`packages/node/README.md`](packages/node/README.md) (npm package).
This file is about _how the codebase works internally_ and _how to develop
on it_, not about how to use the published artifacts.

## 1. What this project is

`mlex` is a from-scratch Rust runtime for running quantized LLMs on Apple
Silicon via [Apple MLX](https://github.com/ml-explore/mlx)'s official `mlx-c`
C API — no Python, no system MLX install, no `mlx-lm`. It loads a Hugging
Face Hub-style model directory (`config.json` + safetensors shards +
tokenizer files) straight onto the GPU through Metal and exposes chat
completion, streaming, tool calling, multi-modal (image/audio/video) input,
reasoning/"thinking" mode, and stateless prompt caching.

It ships as two published artifacts built from one codebase:

- **Rust crate** [`mlex`](https://crates.io/crates/mlex) — the engine.
- **npm package** [`mlex.js`](https://www.npmjs.com/package/mlex.js) — Node.js/TypeScript
  bindings (NAPI-RS), wrapping the same Rust engine.

Everything here targets **macOS on Apple Silicon only** — MLX's Metal
backend doesn't run on Intel Macs, so there is deliberately no x86_64
target anywhere in this repo (build scripts, npm `napi.targets`, CI runners).

### Why it exists / design philosophy

- **No Python at runtime.** `mlx`/`mlx-c` are vendored C++ sources built via
  `build.rs` + CMake and statically linked. `cargo build` (or `npm install`,
  which ships a prebuilt addon) is the whole story.
- **Config-driven, not per-model-hardcoded.** Quantization scheme (affine
  N-bit, mxfp4/mxfp8/nvfp4, OptiQ/QAT mixed precision) is resolved from
  `config.json`'s `quantization` section (`crates/mlex/src/quant.rs`), not a
  model allowlist. Any checkpoint using a wired-up architecture works across
  every quantization scheme, and vice versa.
- **Stateless, OpenAI/Anthropic-style chat API.** There is no
  session/conversation handle (a `Conversation` API existed early on and was
  deliberately deleted — see §6). Callers pass the full message transcript
  on every call; an internal `PromptCachePool` transparently reuses KV
  state for whatever prefix was already computed.
- **One Rust crate.** `mlex-sys` (raw FFI bindings) used to be a separate
  crate; it was folded into `mlex` as a private `sys` module so there's
  exactly one crate to `cargo add` and one thing to publish.

### 3.1 `models/` — architecture dispatch

`Model` is an enum (not a trait object — avoids `dyn`-safety friction from
each architecture needing a different per-layer cache shape):

```rust
pub enum Model {
    Qwen2(qwen2::Qwen2Model),       // also covers Qwen2.5/Qwen2.5-Coder, llama-shaped (MiniCPM5)
    Qwen3(qwen3::Qwen3Model),       // dense, QK-norm
    Qwen35(qwen3_5::Qwen35Model),   // Qwen3.5/3.6, GatedDeltaNet hybrid + MoE variant
    Gemma4(gemma4::Gemma4Model),    // text + optional vision/audio towers
    NemotronH(nemotron::NemotronModel), // hybrid Mamba2/GatedDelta/attention
    Dhara(dhara::DharaModel),       // Canon-conv LLaMA3-style GQA
}
```

`Model::load` reads `config.json`'s `model_type` and dispatches. Adding a
new architecture = a new module here + a new enum arm + a dispatch case in
`models/mod.rs`; new quantization schemes live entirely in `quant.rs` and
don't require touching any architecture code.

| `model_type`                                  | Arm         | Notes                                                                                                                                         |
| --------------------------------------------- | ----------- | --------------------------------------------------------------------------------------------------------------------------------------------- |
| `qwen2`, `llama`                              | `Qwen2`     | Also covers MiniCPM5 and similar vanilla-GQA checkpoints                                                                                      |
| `qwen3`                                       | `Qwen3`     | Dense, per-head QK-norm                                                                                                                       |
| `qwen3_5`, `qwen3_5_moe` (+ `_text` variants) | `Qwen35`    | Dense + MoE (`moe.rs`); most layers use `gated_delta.rs`'s linear-attention recurrence, full attention every `full_attention_interval` layers |
| `gemma4`, `gemma4_text`                       | `Gemma4`    | Text-only and multi-modal variants (see below)                                                                                                |
| `nemotron_h`                                  | `NemotronH` | Interleaves `mamba2.rs` SSM layers, plain attention, and gate-free MLP per `hybrid_override_pattern`                                          |
| `dhara_ar`                                    | `Dhara`     | Canon depthwise convs at up to 4 positions/block, post-RoPE QK-norm (opposite order from Qwen3), logit softcap                                |

`models/cache.rs`'s `LayerCache` enum generalizes per-layer state across all
of these: `KvCache` (growing K/V for quadratic attention), `GatedDeltaCache`
(fixed conv window + recurrent state matrix for linear-attention layers),
and a Canon conv-state variant for DharaAR.

Gemma4 (`models/gemma4/`) additionally carries optional towers:

- `vision.rs` — SigLIP-style ViT + multimodal embedder (`vision_config`).
- `audio.rs` — Conformer-style audio tower (`audio_config`).

Both towers' outputs get spliced into the text embedding stream at the
placeholder token positions the chat template renders (see §3.2).

### 3.2 `media/` — multimodal preprocessing

Independent of any one architecture (though today only Gemma4 checkpoints
ship vision/audio tower weights):

- `image.rs` — resize/rescale matching Gemma4's expected preprocessing
  (soft-token budget `[40, 280]`, `patch_size=16`).
- `audio.rs` — 16 kHz mono PCM → `[T, 128]` log-mel spectrogram, matching
  Gemma4's expected audio feature extraction.
- `video.rs` — uniform frame sampling via a shelled-out `ffmpeg` (file-path
  in, file-based PNG frames out — **not** stdin/stdout piping; piping was
  SIGPIPE-prone for longer clips), then each frame reuses the image path.

`Session::encode_chat_with_media*` (`generate.rs`) renders the chat
template (which emits `<|image|>`/`<|audio|>`/`<|video|>`-style
placeholders), then walks the token stream expanding each placeholder into
the model's actual `boi/boa + soft_token × N + eoi/eoa` span and recording
the preprocessed tensors in placeholder order for the fusion pass.

### 3.3 Reasoning + streaming classification (`reasoning.rs`, `streaming.rs`)

Added to support Qwen3/3.5/3.6, Gemma4, MiniCPM5, and NemotronH's opt-in
"thinking" mode:

- `GenerateOptions::enable_thinking: Option<bool>` threads into the chat
  template as the `enable_thinking` Jinja variable every one of those
  templates checks (`{% if enable_thinking is defined and enable_thinking %}`).
  `None` omits the key (`Undefined`, not `false`) so template `is defined`
  guards see exactly what an un-set kwarg looks like in the reference
  Python/Jinja implementations.
- `reasoning::split_reasoning(text) -> (Option<reasoning>, answer)` finds
  whichever of two known marker-pair conventions appears first —
  Qwen-style `<think>...</think>` or Gemma4-style
  `<|channel>thought...<channel|>` — and splits accordingly. Marker
  detection is deliberately generic/architecture-agnostic rather than
  per-family-hardcoded.
- `reasoning::ReasoningBudget` tracks tokens generated since a reasoning
  span opened and, once `GenerateOptions::reasoning_budget_tokens` is
  exceeded, signals the decode loop to **teacher-force the close marker's
  tokens** through the model (updating KV caches as if the model had
  generated them itself) before resuming normal sampling — mirroring
  Anthropic's extended-thinking `budget_tokens` behavior (cut the thought
  short, move on to the answer) rather than just truncating output.
- `streaming::StreamClassifier` does the same marker-based detection
  incrementally, per streamed token, tagging each `GeneratedToken` with a
  `TokenKind` (`Text`/`Reasoning`/`ToolCall`) so a consumer can render each
  live — mirroring OpenAI/Anthropic's typed streaming deltas. Best-effort
  at token granularity (a marker split across two tokens is only detected
  once both halves have arrived); it retains a small trailing-byte window
  rather than rescanning the whole buffer, so it stays cheap over long
  generations.

**Critical, non-obvious finding** (see §6.5): this marker-based detection
only works if the decode path preserves the marker tokens as literal text.
`Tokenizer::decode`/`decode_piece` use `skip_special_tokens=true` (correct
for normal display), but Gemma4's `<|channel>`, `<channel|>`,
`<|tool_call>`, `<tool_call|>`, `<|think|>` tokens are flagged `special:
true` in its `tokenizer.json`, so that flag silently deletes them —
breaking reasoning/tool-call detection specifically for Gemma4 (Qwen's
equivalent markers are `special: false`, so they worked by accident).
Fixed via `Tokenizer::decode_raw`/`decode_piece_raw`
(`skip_special_tokens=false`), which `reasoning.rs`/`streaming.rs`/the
`generate_cached` final-text computation use internally, while the
eos-token id is explicitly filtered out first so it doesn't leak its
literal spelling into the final answer. **If you add a new
reasoning/tool-call-supporting architecture, check whether its delimiter
tokens are marked `special` in `tokenizer.json`'s `added_tokens` — if so,
you must decode via the `_raw` variants to detect them.**

### 3.4 `prompt_cache.rs` — stateless prompt caching

Mirrors how OpenAI/Anthropic's hosted APIs expose caching: no session
handle; `Session::generate_cached` takes the _full_ message transcript
every call, looks up the longest cached prefix (by token-id sequence) in an
internal `PromptCachePool`, runs only the uncached suffix through the
model, and stores the extended lineage back. Two independent calls sharing
only a common prefix (e.g. the same system prompt) both benefit — this is
the property a single-lineage `Conversation` handle could never have (see
§6.4 for why the old `Conversation` API was removed).

Pool semantics: LRU + TTL eviction, pinning support, and a
`min_cacheable_tokens` gate (default 8) so trivially short prompts aren't
wastefully cached. Callers can opt a single call out entirely via
`GenerateOptions::prompt_cache: Some(false)` (`promptCache` on the Node
binding): the call prefills from scratch and neither reads from nor writes
to the pool.

Pool *sizing* (max entries / TTL / min-cacheable-tokens) is a load-time
setting, not a per-call one: `Session::load_with_cache_config` takes a
`PromptCacheConfig` (`Session::load` just calls it with
`PromptCacheConfig::default()`), and the Node binding's
`MlexModel.load(modelPath, promptCache?)` takes an optional
`JsPromptCacheConfig` (`maxEntries`/`ttlSeconds`/`minCacheableTokens`) as
its second argument. Don't confuse this with `JsGenerateOptions.promptCache`
(the per-call boolean) — same name, different level: one sizes the pool
once when the model is loaded, the other opts one call out of it. Cache reuse leverages MLX's cheap `Array::clone`
(reference-counted, not a deep copy) for the stored KV state.

`generate_cached` returns a `GenerateReply { text, tool_calls, usage,
reasoning }` — not a growing tuple; if you're tempted to add a 5th return
value, convert/extend this struct instead (there's a documented history of
this being a `(String, Vec<ToolCall>)` → 3-tuple → struct progression;
don't regress to tuples).

`Usage { prompt_tokens, cached_tokens, completion_tokens }` mirrors
OpenAI/Anthropic's `usage` block; `cached_tokens` is how many of
`prompt_tokens` were served from the pool.

## 4. Dependencies and vendored libraries

### Vendored (built from source, no system install)

`crates/mlex/vendor/{mlx,mlx-c}` are pinned upstream checkouts (own `.git`
stripped) — see `crates/mlex/vendor/PINS.md` for exact commit hashes and
how to refresh them. `crates/mlex/build.rs`:

1. Builds `mlx-c` via CMake, pointing `FETCHCONTENT_SOURCE_DIR_MLX` at the
   vendored `vendor/mlx` (so CMake doesn't try to clone MLX itself).
2. Links `libmlxc`+`libmlx` statically, plus macOS frameworks (Metal,
   MetalPerformanceShaders, Foundation, QuartzCore, Accelerate) and `c++`.
3. Locates the built `mlx.metallib` and exposes its path via
   `cargo:rustc-env=MLEX_METALLIB_PATH` for anything that needs to colocate
   it (MLX also bakes an absolute build-tree path in as a fallback).
4. Runs `bindgen` over `mlx-c`'s `mlx/c/mlx.h`, allow-listing `mlx_*`/`_mlx_*`
   symbols, into `OUT_DIR/bindings.rs` (consumed by `src/sys.rs`).

Build requirements: Xcode Command Line Tools (C++17 + `libclang`), CMake.
First build compiles all of MLX/mlx-c from source (slow, several minutes);
subsequent builds are incremental like any Rust crate as long as
`vendor/**` doesn't change (CI keys its cache on
`hashFiles('crates/mlex/vendor/**')`).

### Crates.io dependencies (`crates/mlex/Cargo.toml`)

`thiserror`, `serde`/`serde_json`, `half` (bf16/fp16), `tokenizers` (HF
tokenizer.json), `minijinja`+`minijinja-contrib` (Jinja2 chat templates,
with a `pycompat` shim for `.startswith()`/`.strip()`-style Python string
methods HF templates lean on), `rand` (sampling), `image` (JPEG/PNG decode,
`rayon` feature), `rustfft` (mel-spectrogram FFT for audio preprocessing).
Build-only: `bindgen`, `cmake`.

### Node side (`crates/mlex-node/Cargo.toml`, `packages/node/package.json`)

`mlex-node` depends on `mlex` (path dependency) + `napi`/`napi-derive` (v3,
features `napi10`, `async`, `tokio_fs`, `serde-json`). It's built with
`crate-type = ["cdylib"]` and `publish = false` — see §6.2 for why it's
never published to crates.io. `packages/node` uses `@napi-rs/cli` to invoke
the actual build (`napi build`) and `vitest` for tests.

## 5. Testing

**Never run the Rust or Node integration/test suites unscoped** (plain
`cargo test -p mlex` or `npm test` with a large/full `models/` directory
present). Both discover and run against _every_ downloaded model by
default (§5.1/§5.2), and real checkpoints are multi-GB Metal/GPU forward
passes — an unscoped run against several checkpoints can run for a very
long time (effectively hanging from the caller's perspective) and/or
exhaust all available unified memory, taking down the machine. Always
point `MLEX_MODELS_DIR` at a directory containing symlinks to only the
specific model(s) the task actually needs before running either suite —
see the scoped-run pattern in §5.4 and the commands in §7. If you only
need fast unit coverage with no models at all, use `cargo test -p mlex
--lib`.

### 5.1 Rust (`crates/mlex/tests/`)

```bash
cargo test -p mlex                                    # unit + integration, full local registry
cargo test -p mlex --lib                               # unit tests only (fast, no models needed)
MLEX_MODELS_DIR=/path cargo test --release -p mlex -- --test-threads=1   # scoped model set
```

Integration suites (each a separate binary under `tests/`):

- `generate_basic.rs` — plain single-shot generation.
- `generate_sampling.rs` — seeded reproducibility, top-k/top-p behavior.
- `multi_turn.rs` — growing message history across `generate_cached` calls.
- `prompt_cache.rs` — numerically proves cached vs. from-scratch recompute
  produce byte-identical output (the pool's core correctness contract).
- `tool_calling.rs` — schema rendering + Hermes/Gemma call parsing.
- `system_prompt.rs` — leading `role: "system"` message support.
- `multi_modal.rs` — image/audio/video input against real Gemma4 checkpoints
  and `samples/*`.
- `architectures.rs` — cross-architecture smoke coverage.

All of them discover models the same way, via `tests/common/mod.rs`'s
`registry()`:

- Scans `MLEX_MODELS_DIR` (default `<repo>/models`) for
  `models--<org>--<name>/snapshots/<rev>/` directories with a
  `config.json`.
- **Classifies CI-eligibility by measured peak RSS, not disk size** — disk
  size is unreliable (NemotronH's Mamba/GatedDelta recurrences upcast to
  f32 and use substantially more runtime memory than their checkpoint size
  implies). It shells out to the `measure_memory` example (must be built
  first: `cargo build --release --example measure_memory -p mlex`), which
  loads the model + runs one short forward pass and reports
  `getrusage(RUSAGE_SELF).ru_maxrss` from a fresh subprocess (so
  measurements don't accumulate across models in one test run).
  Measurements are cached in `target/mlex-test-cache/model-memory.json`,
  keyed by `"{dir}|{weights_bytes}"` (so a re-download invalidates it).
- A model is `ci_safe` if measured peak RSS ≤ `MLEX_MAX_MODEL_GB` (default
  `5.0`). Only `ci_safe` models are returned unless
  `MLEX_INCLUDE_LARGE_MODELS=1` is set (CI never sets this) — this is how
  the _same_ test suite runs unattended in CI against small models and,
  locally, against anything you've downloaded, including
  multi-billion-parameter checkpoints too large for a GitHub-hosted runner.
- `registry_for_family(model_type)` and `has_capability(dir, "vision_config"
| "audio_config" | ...)` let tests target specific architectures/modalities
  without a hardcoded model list.

Run with `--test-threads=1` when exercising real models: MLX's Metal
context is thread-local (see §6.1) and, more importantly, running multiple
multi-GB models' forward passes concurrently on a single GPU/unified-memory
budget is how you OOM a CI runner.

### 5.2 Node (`packages/node/test/`)

```bash
cd packages/node
npm run build   # regenerates the native addon + index.js/index.d.ts — do this
                # after ANY change to crates/mlex-node/src/lib.rs or crates/mlex
npm test        # vitest run
npm run test:coverage
```

`test/models.ts` mirrors `tests/common/mod.rs` exactly but does **not**
re-measure memory — it reads the same
`target/mlex-test-cache/model-memory.json` the Rust suite writes. This
means **the Rust suite must run before the Node suite** in any environment
that hasn't already populated that cache (CI's `ci.yml` does this
ordering; keep it if you touch the workflow). If the cache file doesn't
exist, Node-side models are conservatively treated as unmeasured/excluded
rather than guessed at.

Test files use kebab-case (`multi-turn.test.ts`, not `multiTurn.test.ts`)
by convention — matches the rest of the npm package's style.

### 5.3 Coverage

`scripts/coverage-rust.sh [--all]` wraps `cargo llvm-cov`. On a
Homebrew-installed rustc (common for local dev on macOS), the Xcode
Command Line Tools' bundled LLVM version usually doesn't match what rustc
was built against, so the script tries `rustup component add
llvm-tools-preview` first (works in CI / rustup-managed toolchains) and
falls back to pointing `LLVM_COV`/`LLVM_PROFDATA` at Homebrew's `llvm` keg
otherwise. `--all` includes integration tests (loads real models); without
it, only unit tests run (fast, no models).

### 5.4 What NOT to do when testing locally

- Don't run the full integration suite (Rust or Node) against every
  downloaded model by default — always scope `MLEX_MODELS_DIR` to a
  directory containing symlinks to just the model(s) you need (see the
  pattern used throughout this repo's history: a temp dir with one
  symlinked Gemma4 checkpoint, e.g. `/tmp/mlex-gemma4-only/`). An unscoped
  run against a full local `models/` directory can appear to hang for a
  very long time and/or exhaust all available memory (see the warning at
  the top of §5).
- Don't assume a model's disk size predicts its test eligibility — always
  go through `registry()`/`registryForFamily()`, never hardcode a path.
- Don't forget `npm run build` after touching `mlex-node`/`mlex` Rust code
  that affects the Node-visible surface — `index.js`/`index.d.ts` are
  generated artifacts, edits to them by hand will be silently clobbered
  (and drift from reality until the next build anyway).

## 6. Key findings, decisions, and their history

These are non-obvious things discovered the hard way; re-reading this
section before touching related code will save time.

### 6.1 MLX streams are thread-local

`mlx-c` requires a `Stream(gpu, 0)` to be registered per-thread; running
`cargo test` with the default multi-threaded test runner produced
`Mlx("no Stream(gpu, 0) in current thread")` errors. Fixed by
`stream.rs`'s `thread_local! { RefCell<Option<StreamHolder>> }` pattern —
don't "simplify" this into a single global stream, and remember
`--test-threads=1` is still advisable for real-model integration tests for
memory reasons even though the thread-local fix handles correctness.

### 6.2 Why `mlex-node` is never published to crates.io

It's `crate-type = ["cdylib"]` — a dynamic library, not an `rlib`, so
there's nothing for another Rust crate to link against even if you
depended on it. Every `#[napi]`-annotated type exists purely to describe a
JS-facing shape; none of it is a meaningful Rust API. `Cargo.toml` sets
`publish = false`. Only its compiled `.node` binary ships, inside the npm
package. See `crates/mlex-node/README.md` for the full rationale if you
need to explain this to someone.

### 6.3 `mlex-sys` → `mlex::sys`

There used to be three crates (`mlex-sys`, `mlex-core`, `mlex-node`). The
raw FFI bindings crate (`mlex-sys`) was merged into `mlex-core` (renamed to
just `mlex`) as a private `sys` module, because `mlex-sys` had no
standalone value — nobody would `cargo add mlex-sys` on its own. This
merge is why `sys.rs` has `#![allow(dead_code)]`: as a private module some
FFI bindings are unused without triggering warnings, which is expected
for a bindings layer that doesn't need every symbol exercised.

### 6.4 The `Conversation` API was deleted on purpose

An earlier design had a stateful `Conversation` handle (own the KV cache,
own the message history). It was removed entirely in favor of the
stateless `generate_cached` + `PromptCachePool` design (§3.4) because a
single-lineage handle can't share cache benefit across independent calls
that happen to share a prefix (e.g. two different users hitting the same
system prompt) — the OpenAI/Anthropic-style stateless model strictly
dominates it. **Do not reintroduce a `Conversation`/session-handle API** —
if you find yourself wanting one, the answer is almost certainly "grow the
message list yourself and call `generate_cached` again," which is the
whole point.

### 6.5 `skip_special_tokens` silently breaks marker-based parsing (see §3.3)

The most subtle bug found in this codebase. Some model families implement
reasoning/tool-call delimiters as literal (non-special) added tokens
(Qwen: `<think>`, `</think>`, `<tool_call>`, `</tool_call>` are all
`special: false`), while others use genuinely special tokens for the same
purpose (Gemma4: `<|channel>`, `<channel|>`, `<|tool_call>`, `<tool_call|>`,
`<|think|>` are all `special: true`). `Tokenizer::decode`'s
`skip_special_tokens=true` (correct for normal display text) silently
deletes the latter, so any string-marker-based parsing over `decode()`'s
output only works for the former by coincidence. Always use
`Tokenizer::decode_raw`/`decode_piece_raw` for marker detection, and only
use the plain `decode`/`decode_piece` for genuinely user-facing display
text. When adding support for a new architecture's reasoning or tool-call
convention, **check `tokenizer.json`'s `added_tokens[].special` for its
delimiter tokens before assuming either decode path works**.

### 6.6 CI model set is deliberately minimal (for now)

`scripts/ci-download-models.sh` only downloads
`mlx-community/gemma-4-E2B-it-qat-4bit` — the smallest available Gemma4
checkpoint. This was a deliberate constraint during initial multi-modal
development ("only run rust/node tests for the smallest Gemma4 model due
to memory constraints") and hasn't been revisited since multimodal
stabilized. If you add coverage for other architectures that should also
run in CI, widen `MODELS` in that script (each addition adds to CI
runtime, bandwidth, and the `hf-models-*` cache key's invalidation
surface) — check current measured RSS via the local `registry()`/`npm
test` cache first to confirm it's actually CI-safe (`≤ 5 GB` peak RSS)
before adding it.

### 6.7 Peak-RSS measurement runs in a subprocess, not in-process

`tests/common::measure_peak_rss` shells out to the `measure_memory`
example rather than measuring in-process, specifically so measurements
across different models in one `cargo test` run don't accumulate (loading
model A then model B in the same process would report B's measurement as
"A + B"). If you change `measure_memory.rs`'s example, remember its stdout
contract (`registry()` parses the trimmed stdout as a bare `u64` byte
count) — don't add extra printed output there.

### 6.8 Video frame extraction avoids stdin/stdout piping

`media/video.rs` shells out to `ffmpeg` using file-path input and
file-based PNG output, not pipes — piping was found to be SIGPIPE-prone
for clips longer than ~2 seconds. If you touch video preprocessing, keep
the file-based I/O pattern.

### 6.9 The npm package must ship `mlx.metallib` next to the addon

MLX locates its compiled Metal kernel library at runtime via the absolute
path baked in at compile time (the cargo build tree — absent on end-user
machines), falling back to a path colocated with the loaded binary. The
first published `mlex.js` (0.1.0) omitted the metallib and every load
failed with "Failed to load the default metallib" on machines without a
local build tree. `packages/node/copy-metallib.mjs` (chained into both
`npm run build` variants) copies the freshest
`target/**/build/mlex-*/out/lib/mlx.metallib` into the package directory,
and `package.json`'s `files` includes `*.metallib`. Don't remove either
half.

### 6.10 Tool-call payloads and value spellings vary per template, not per family

Three findings from wiring real mlx-community checkpoints into an agent
harness:

- Newer Qwen3.5/NemotronH templates render tool calls as
  `<tool_call><function=NAME><parameter=KEY>VALUE</parameter>...</function></tool_call>`
  (XML-function style), not Hermes JSON — `parse_hermes_payload` tries
  JSON first and falls back to the XML parser. XML parameter values are
  coerced through a JSON parse when they form a complete JSON value and
  stay plain strings otherwise.
- Gemma4 templates wrap string argument values in a dedicated special
  quote token whose raw decode is `<|"|>` (marker parsing runs over raw
  decodes per §6.5, so it survives into the parser) —
  `parse_gemma_scalar` normalizes it to a plain quote.
- Streamed marker remnants: markers whose spelling is (partly) ordinary
  tokens leak their display text into the token stream — Qwen's
  `<think>`/`</think>` are non-special (display decode keeps them), and
  Gemma4's `<|channel>thought` open marker ends in the ordinary word
  `thought`. `StreamClassifier::last_marker()` reports the marker a
  `classify` call just completed, and `generate.rs`'s
  `stream_display_text` suppresses that token's remnant so streaming
  consumers get clean span content (mirroring OpenAI/Anthropic typed
  deltas, which never include wire markers).

### 6.11 Reasoning budget force-close is teacher-forced, not truncated

`ReasoningBudget`'s force-close (§3.3) doesn't just stop generation early
when the token budget is hit — it encodes the close marker text back to
token ids and runs them through `Model::forward` as if the model had
generated them (updating KV caches), then resumes normal sampling for the
actual answer. This is deliberately more involved than truncation because
truncating mid-thought would leave generation in a "still inside the
reasoning span" cache state with no answer ever produced; forcing the
close and continuing is what makes `reasoning_budget_tokens` produce a
_usable_ answer rather than just a shorter, incomplete thought.

## 7. Quick reference: "how do I test X?"

| You want to...                              | Do this                                                                                                                                                          |
| ------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Run all fast unit tests (no models)         | `cargo test -p mlex --lib`                                                                                                                                       |
| Run everything against local models         | `cargo test --release -p mlex -- --test-threads=1`                                                                                                               |
| Run against one specific model only         | `MLEX_MODELS_DIR=/tmp/some-dir cargo test --release -p mlex -- --test-threads=1` (populate the dir with one symlinked `models--org--name` entry)                 |
| Include large, non-CI-safe local models     | add `MLEX_INCLUDE_LARGE_MODELS=1`                                                                                                                                |
| Change the CI-eligibility memory ceiling    | `MLEX_MAX_MODEL_GB=<n>`                                                                                                                                          |
| Test one architecture family                | filter via `common::registry_for_family("gemma4_text")` etc. inside the test, or just point `MLEX_MODELS_DIR` at a dir with only that family                     |
| Test reasoning/streaming end-to-end quickly | `cargo run --release -p mlex --example reasoning -- <model_dir>`                                                                                                 |
| Test raw generation/logits debugging        | `cargo run --release -p mlex --example generate -- <model_dir>` (supports `MLEX_DEBUG_LOGITS`/`MLEX_DEBUG_LAYERS` env vars for layer-by-layer numeric debugging) |
| Rebuild the Node addon after a Rust change  | `cd packages/node && npm run build` (regenerates `index.js`/`index.d.ts` too)                                                                                    |
| Run the Node suite                          | `cd packages/node && npm test` (run the Rust suite first if `target/mlex-test-cache/model-memory.json` doesn't exist yet)                                        |
| Get Rust coverage                           | `./scripts/coverage-rust.sh --all`                                                                                                                               |
| Get Node coverage                           | `cd packages/node && npm run test:coverage`                                                                                                                      |

## 8. Conventions to preserve

- **No x86_64 anywhere.** Not in `Cargo.toml` targets, not in
  `napi.targets`, not in CI runners, not in docs.
- **`generate_cached` returns a struct (`GenerateReply`), not a tuple.**
  Extend it, don't revert to positional tuples.
- **Test files under `packages/node/test/` use kebab-case.**
- **`GenerateOptions`/`JsGenerateOptions` fields should stay `Option<T>`
  with template-`Undefined`-preserving semantics** where the underlying
  Jinja template distinguishes "not passed" from "explicitly false/None" —
  see how `tools` and `enable_thinking` are threaded into
  `apply_chat_template_full`'s context for the pattern to follow for any
  new template-conditional feature.
- **Don't hand-edit `packages/node/index.js`/`index.d.ts`** — they're
  `napi build` output.
- **Marker-based text parsing (reasoning/tool-calls) must decode via the
  `_raw` variants**, per §6.5.
- **Prefer measured peak RSS over disk size** for any new CI-eligibility
  logic — see §6.6/§6.7.
